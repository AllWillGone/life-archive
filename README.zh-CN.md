# LifeArchive

[English](README.md) | 简体中文

LifeArchive 是一个用于 AI 辅助长期记忆维护的自传式记忆系统模板。它把事实、情绪、会话摘要、待归档碎片和可选支持记录分开保存，让私人记忆库在长期使用中仍然保持清晰、可追溯、可维护。

本仓库只包含规则、目录结构、空模板和去标识化示例，不应保存真实个人记忆。实际使用时，建议复制为私人仓库；公开分享时，应发布空模板或经过彻底清理的副本。

## 目录职责

- `summary/` 用来快速同步整体情况。
- `fact/` 按完整事件记录事实，以事件开始时间和事件名排序。
- `feeling/` 记录使用者对同一事件在不同时间节点的情绪、看法和变化。
- `session/` 只保存每次对话后的记忆更新摘要，不保存完整聊天记录；普通建议、行动提醒、安慰话术和临场分析默认只在这里简短保留。
- `support/` 是可选的长期支持层；只有当使用者明确希望长期回看建议、行动提醒或支持性分析时，才单独归档。
- `inbox/` 暂存暂时无法归档的事实碎片、关系线索或事件线索。
- `archive/` 保存旧格式、废弃内容或迁移前备份。

维护这个仓库的 AI assistant 应先阅读 `AGENTS.md`，再按摘要到细节的顺序读取记忆。`AGENTS.md` 中包含维护规则、目录边界和推荐读取顺序；`docs/templates/` 提供可复制的空模板。

## 核心原则

- 事实与情绪分开：`fact/` 记录发生了什么，`feeling/` 记录使用者如何感受、理解和变化。
- 底层记忆优先保护：整理时不应为了压缩、润色或概括而改写已有事实、细节、情绪和原始表述。
- 会话摘要不等于聊天记录：`session/` 只保存对话后的记忆更新摘要，不保存完整聊天。
- 建议类内容默认不进入底层记忆：普通建议、行动提醒、安慰话术和临场分析默认只在当天 `session` 中简短保留；只有使用者明确希望长期回看时，才使用可选 `support/`。
- 信息不足时先暂存或标注不确定，避免编造、过度推断或为了整理而强行归档。

## 新对话启动提示词

每次重新开启对话时，可以先发送。请把 `<你的 LifeArchive 路径>` 替换成实际路径；如果 agent 已经在仓库目录中工作，也可以直接写“本仓库的 AGENTS.md”。

适合：普通聊天时，让 agent 先理解记忆系统，再自然回应并轻量维护记忆。

不适合：大规模整理、合并、拆分或压缩文件。

```text
请先按 <你的 LifeArchive 路径>\AGENTS.md 读取我的记忆系统，然后像平常一样优先和我聊天、分析和回应我。记忆维护不要压过正常回应：当对话中出现新的事实、情绪变化、关系线索或事件线索时，在本轮中按规则安静更新 <你的 LifeArchive 路径> 下的记忆文件；能归入完整事件的写入对应事件，暂时无法准确归档的先写入 inbox。agent 主动给出的建议、行动提醒、安慰话术或支持性分析默认不要写入 fact/feeling/summary，也不要另行创建长期 support 索引；普通建议只在当天 session 摘要中简短保留。只有当建议已经变成我明确认可、复述、采纳，或表达为“我想 / 我需要 / 我希望”的内容时，才可作为我的需要、愿望、决定或当时理解写入 feeling 或必要的 summary/current。默认不要在回复结尾主动汇报“已记录”或“已更新”，除非我明确询问、更新涉及重要结构变化，或本轮任务本身就是整理记忆。若需要说明更新情况，必须先确认文件实际写入成功，不能凭计划或印象声称已经更新。
```

## 周期整理提示词

每隔一段时间，或发现索引、排序、摘要开始变乱时，可以发送：

适合：阶段性体检整个记忆系统，处理索引滞后、事件过碎、重复记录、归档边界混乱、链接或排序问题。

不适合：不用于日常聊天；也不适合只想轻量压缩 `summary/` 的时候。

```text
请按 <你的 LifeArchive 路径>\AGENTS.md 读取并整理我的自传式记忆系统。这次任务是周期整理，不是普通聊天；但仍要遵守对话优先、真实性、写入确认、底层记忆保护和事实/情绪分离规则。fact/ 和 feeling/ 里的既有内容是我的底层记忆，我通常不会再完整复述；整理时不要为了压缩、润色或概括而删除、弱化、替换、改写已有事实、细节、情绪和原始表述。原则上只做时间顺序调整、原文搬移、补链接、补日期标注和少量必要语言串联；不确定能否删改时，先保留并询问我。

请先只审计，不要写入。按推荐读取顺序检查：summary/overview.md、summary/current.md、summary/timeline.md、fact/index.md、相关 fact/events/、feeling/index.md、相关 feeling/by_event/、summary/people.md、summary/topics.md；如果已经启用可选 support 长期支持层，再检查 support/index.md 和相关 support/by_event/；随后检查 inbox/unresolved.md，以及最近的 session 文件。

重点检查并提出方案：
1. fact 与 feeling 是否混写；是否有事实应迁移到 fact，情绪和看法变化应迁移到 feeling。
2. fact/events 是否按完整事件、关系阶段或人生阶段组织；是否有重复事件、过碎事件、该合并的小事，或聚合事件已经可以拆分。特别检查新建 fact 文件边界：只有当内容有独立对象、独立阶段、持续发展、长期意义，或确实无法归入已有事件时，才应保留为独立 fact；普通琐事、一次性聊天、短期情绪波动和已有事件的补充，优先并入已有 fact、当天 session 或 inbox。
3. fact/index.md、feeling/index.md、summary/timeline.md 和事件内 ## 时间线 是否按事件开始时间或条目发生时间排序；回溯补充的信息是否被错误追加到末尾。
4. 修改过的 fact/events 和 feeling/by_event 文件内“最近更新”是否滞后。
5. summary/current.md 是否过长、过细或像事件合集；过期内容是否应沉淀到事件文件和 summary/timeline.md。
6. summary/overview.md、summary/people.md、summary/topics.md 是否需要补充、压缩、去重或修正链接。
7. inbox/unresolved.md 中是否有碎片已经可以归档；是否有不确定信息应该转入对应事件的“不确定或待澄清”部分。
8. 如果启用了可选 support 长期支持层，检查 support/ 是否只保存建议、行动提醒和支持性分析；如果没有启用，则检查是否仍有 agent 建议、行动提醒、安慰话术或支持性分析误留在 fact/feeling/summary 中。只有已经变成我明确认可、复述、采纳或表达为“我想 / 我需要 / 我希望”的内容，才可保留在 feeling 或 summary/current。
9. session 文件是否只记录本次对话后的记忆更新摘要，没有保存完整聊天记录；是否需要为本次整理创建或更新当天 session 摘要。
10. 链接、文件名、日期精度、真实姓名、推测/不确定标注和 UTF-8 读写是否一致。

输出时先给我一个简短整理方案：列出建议修改的文件、每类问题的处理办法、是否有需要我确认的拆分/合并/删除/归档决定。等我确认后，再实际写入文件。写入完成后，必须基于实际成功的文件操作简短说明更新了哪些文件；如果有任何写入失败或不确定，明确说明尚未写入成功。
```

## 阶段性摘要瘦身提示词

适合：`summary/` 逐渐变长、变细，或者未来 agent 读取摘要后容易被过多近期细节带偏时使用。

不适合：不用于拆分/合并底层事件，也不用于改写 `fact/` 和 `feeling/` 的原始记录。

```text
请按 <你的 LifeArchive 路径>\AGENTS.md 读取我的记忆系统。这次任务是“阶段性摘要瘦身”，不是完整周期整理，也不是普通聊天。

目标：让 summary/ 下的入口文件更轻、更像快速导航，避免它们变成事实和情绪底层记录的重复副本。请优先保护 fact/ 和 feeling/ 中的底层记忆，不要改写、删除、压缩或替换底层原文。除非发现明显断链、过期链接或摘要引用错误，否则不要修改 fact/ 和 feeling/。

请重点检查并优化：
1. summary/current.md：只保留仍影响当前状态的活跃关系、近期困扰、当前目标、情绪主线和确实需要跟进的事项。已经沉淀到 fact/events、feeling/by_event 或 summary/timeline.md 的细节，应从 current 中压缩或移除。
2. summary/timeline.md：只做摘要时间线，每条尽量是一句核心事实 + 相关链接。不要在 timeline 中复述完整事件过程、复杂情绪分析或大量细节。
3. summary/people.md：只记录人物与我的关系定位、当前仍重要的关系状态和相关事件入口。不要写成他人的小传，也不要重复 fact 文件里的详细经过。
4. summary/topics.md：只保留主题入口和相关事件链接。不要把主题索引写成新的主题性长文。
5. 如果发现 summary 中有 agent 建议、行动方案、安慰话术或临场分析，除非已经明确变成我的想法、需要、决定或当前状态，否则不要保留在 summary 中。
6. 如果某条摘要内容仍有长期意义，但不适合留在 summary，请确认它已经存在于对应 fact/feeling/session 中；如果没有，先提出迁移方案，不要直接删除。

处理原则：
- 只瘦身 summary 层，默认不动 fact/ 和 feeling/。
- 保留信息入口，不保留完整细节。
- 优先压缩、合并和移除重复摘要，不改变事实原意。
- 不确定某条内容是否还能删时，保留并标注为“待确认”，或先问我。
- 修改前先给我一个简短方案，列出准备改哪些 summary 文件、每个文件怎么瘦身、有没有需要我确认的内容。等我确认后再写入。
- 写入完成后，检查链接、标题、更新时间和是否仍有明显重复；最后基于实际写入结果说明更新了哪些文件。
```

## 开源与隐私提醒

LifeArchive 面向真实、长期、细节化的自传式记录。填充后的个人记忆仓库可能包含姓名、地点、关系、情绪、会话摘要和 Git 历史中的旧版本信息。不要直接公开真实记忆仓库；公开时应使用本模板或经过彻底清理的新仓库。
